# AI 驱动构建具体成果文档

## 项目名称

KOReader TXT 自动目录插件（`txtautotoc.koplugin`）

## 项目概述

本项目通过 AI 辅助完成一个独立 KOReader 插件，用于在 Kindle / KOReader 打开 TXT 小说时自动识别章节标题，并将生成结果接入 KOReader 默认“目录 / Table of contents”入口。插件重点解决中文 TXT 小说没有目录、章节跳转不准、GB18030 编码无法识别、首次打开卡顿和设备端调试困难等问题。

最终交付物包括：

- 可直接安装到 KOReader 的 `txtautotoc.koplugin` 插件目录。
- 中文默认 README、英文 README、中文独立说明文档。
- 解析器、映射器、缓存模块、兼容读取模块和 GB18030 内置解码模块。
- Luajit 单元测试，覆盖解析、缓存、映射、主插件流程和编码读取。
- Kindle Paperwhite 5 越狱版设备路径下的实机插件同步与验证。

## 1. 项目解决的核心痛点

### 1.1 TXT 小说缺少可用目录

大量中文 TXT 小说没有内置 TOC，KOReader 默认目录入口无法直接展示章节。用户只能通过全文搜索、进度条或手动翻页定位章节，阅读长篇小说时效率很低。

本插件自动扫描 TXT 正文中的章节标题，生成目录结构，并接入 KOReader 原生目录入口，使用户无需学习新的入口或操作习惯。

### 1.2 中文章节标题格式不统一

中文小说常见标题包括 `第1章`、`第一百章`、`第N卷`、`序章`、`番外`、`尾声` 等，也存在广告前缀、Markdown 标题、英文 `Chapter / Part` 等混合格式。单一正则容易误判正文或漏判章节。

插件将章节识别拆成规则化解析器，按中文、英文、Markdown、特殊标题和卷/章层级分别处理，并设置“至少 3 个有效章节才接管目录”的安全阈值，降低误识别风险。

### 1.3 目录点击跳转不准

早期方案如果只按行数估算页码，目录能显示但点击章节可能跳到附近页面，而不是章节开头。用户反馈“点击访问章节之后，进度没有正常跳转到章节的开头”。

插件后续加入批量精确定位逻辑：对识别出的标题调用 KOReader 文本搜索能力获取真实 xpointer，再注入目录。这样目录项跳转更接近章节标题位置。

### 1.4 首次打开大 TXT 容易卡顿

长篇 TXT 小说可能有数百章、数万行。逐章搜索会造成 Kindle 打开书籍时明显卡顿。

插件采取两级策略：

- 打开时优先使用批量精确搜索，减少重复全文扫描。
- 如果精确映射不足，退回快速页码估算，保证目录可用。
- 每本书按文件路径、mtime、size 和检测器版本缓存结果，文件未变化时直接复用缓存。

### 1.5 GB18030 / GBK 编码文件无法识别

用户导入的一批中文 TXT 文件在 macOS 上检测为 GB18030。插件最初直接按原始字节解析，导致中文标题变成乱码，章节识别结果为 0。

修复后插件读取层会先判断 UTF-8；如果不是合法 UTF-8，则按 GB18030/GBK 转成 UTF-8 再解析。针对 Kindle 上没有可用 `iconv` 的情况，项目新增内置 GB18030 解码表，避免依赖设备系统命令。

### 1.6 设备端问题难排查

插件在电脑上测试通过后，Kindle 上仍可能因为 KOReader 版本、系统命令缺失、插件路径错误、缓存状态或旧 metadata 导致行为不同。

本项目建立了设备侧排查路径：

- 检查 `/Volumes/Kindle/koreader/plugins/txtautotoc.koplugin` 文件完整性。
- 读取 `/Volumes/Kindle/koreader/crash.log`。
- 检查书籍 `.sdr/metadata.txt.lua` 中的 `txtautotoc_last_status` 和 `txtautotoc_last_encoding`。
- 用同一批 Kindle 卷 TXT 文件在本机复现解析结果。

## 2. 核心逻辑流

### 2.1 自动生成目录主流程

```text
打开 TXT 书籍
  -> KOReader 触发 onReaderReady
  -> 插件判断是否为 TXT
  -> 判断全局开关是否启用
  -> 判断是否已有 KOReader 手工目录
  -> 构建缓存签名：file path + mtime + size + detector_version
  -> 缓存命中：直接注入目录并刷新 TOC
  -> 缓存未命中：读取 TXT 文件
  -> 识别编码并转成 UTF-8
  -> 章节解析器扫描标题行
  -> 映射章节位置
  -> 有效章节数 >= 3：注入 document.getToc 包装器
  -> 触发 UpdateToc
  -> 保存缓存和状态
```

该流程的关键点是“只在满足条件时接管目录”。如果不是 TXT、插件关闭、存在手工目录、有效章节不足，插件都会退出并保留 KOReader 原有行为。

### 2.2 编码兼容读取流程

```text
读取 TXT 原始字节
  -> 判断是否为合法 UTF-8
  -> 是：直接返回 UTF-8 文本
  -> 否：尝试系统 iconv 转 GB18030/GBK -> UTF-8
  -> iconv 不可用或失败：使用插件内置 GB18030 解码表
  -> 解码成功：返回 UTF-8 文本，并记录 txtautotoc_last_encoding = "gb18030"
  -> 全部失败：返回 raw，并记录 raw 状态用于排查
```

这个流程解决了 Kindle 设备端与电脑环境不一致的问题。实际排查中，Kindle metadata 显示 `txtautotoc_last_encoding = "raw"`，说明设备上 `iconv` 不可用；新增内置解码后，同一批 GB18030 TXT 可在禁用 `iconv` 的情况下继续识别章节。

### 2.3 章节解析流程

```text
UTF-8 文本
  -> 去除 BOM
  -> 标准化 CRLF / CR 换行
  -> 按行扫描
  -> 跳过开头伪目录页
  -> 识别 Markdown / 中文卷部 / 中文章节 / 特殊标题 / 英文标题
  -> 根据卷、部、篇、章、回、节计算 depth
  -> 生成候选目录项
```

解析器只接受“独占一行”的标题，避免正文中出现“第 N 章”字样时被误判为目录项。

### 2.4 章节位置映射流程

```text
候选目录项
  -> 批量组织搜索关键词
  -> 调用 KOReader document.findAllText
  -> 将搜索结果映射回章节标题
  -> 去除重复标题和非单调递增结果
  -> 得到真实 xpointer / page
  -> 注入 TOC
```

目录项不直接使用文件字节偏移，而是使用 KOReader 文档对象搜索得到的 xpointer。这样可以适配 KOReader 对 TXT 的分页、重排和内部文档结构。

### 2.5 菜单和人工控制流程

插件在阅读器菜单中提供 `TXT 自动目录`：

- `启用自动生成`
- `打开时精确定位`
- `立即生成/重建目录`
- `清除当前书籍缓存`
- `显示通知`

当用户手动点击 `立即生成/重建目录` 时，插件会强制重建当前书籍目录；当用户点击 `清除当前书籍缓存` 时，会移除缓存、状态和编码记录，下次打开或重建时重新生成。

## 3. 是否包含长链推理

本项目包含明显的长链推理过程。

原因不是单纯“写一个正则”，而是需要跨多个层面持续推导：

- 从用户现象“TXT 无法识别目录”追踪到章节解析输入是否为乱码。
- 从本机能识别、Kindle 不能识别，进一步定位到设备端 `iconv` 不可用。
- 从 KOReader 默认目录不显示，追踪到 `document.getToc()` 包装器、`UpdateToc` 事件和 `.sdr/metadata.txt.lua` 状态。
- 从“点击目录不跳到章节开头”，推导出页码估算不足，需要改为基于 KOReader 搜索结果的 xpointer 映射。
- 从“打开书籍卡住”，推导出逐章全文搜索成本过高，需要批量搜索、缓存和进度反馈。
- 从“已有手工目录不能被覆盖”，推导出插件接管逻辑必须有优先级保护。

这些问题之间存在依赖关系：编码不对会导致解析失败，解析失败会导致目录不足，映射不准会导致跳转不准，缓存不失效会导致旧错误持续存在。因此构建过程需要连续分析、验证和回溯，而不是一次性生成代码。

## 4. 是否包含多 Agent 协作

本项目实际构建过程中没有启动独立的多 agent 并行协作。

采用的是“单 AI Agent + 工具链 + 测试闭环”的方式完成：

- 单 Agent 负责需求拆解、代码阅读、实现、测试、设备排查和文档更新。
- 使用并行工具调用加速文件检查、日志读取、checksum 对比和测试验证。
- 使用测试驱动方式验证关键行为，例如 GB18030 在无 `iconv` 环境下仍能解码。
- 使用系统化调试方式从 Kindle metadata 和 crash log 中定位真实根因。

如果后续要改造成多 Agent 协作，可以拆成以下角色：

- 解析规则 Agent：专注章节标题规则、误判样本和单元测试。
- KOReader 集成 Agent：专注插件生命周期、菜单、TOC 注入和 xpointer 跳转。
- 设备排障 Agent：专注 Kindle 日志、metadata、性能和兼容性。
- 测试验证 Agent：专注 fixture、回归测试、真实书籍批量验证。
- 文档发布 Agent：专注 README、使用说明、成果文档和宣发材料。

当前阶段没有引入多 Agent 的主要原因是项目规模仍可由单 Agent 串行掌控，且 KOReader 插件文件之间耦合较高，并行修改容易增加冲突和集成成本。

## 5. AI 驱动构建方式

### 5.1 需求驱动

项目从用户真实使用问题出发，不预设完整方案，而是按反馈持续演进：

- 先解决“TXT 自动目录生成”。
- 再解决“中文按钮和中文文档”。
- 再解决“打开书籍卡住”。
- 再解决“目录点击跳转不准”。
- 再解决“GB18030 文件无法识别”。
- 最后针对 Kindle 实机环境补内置解码。

### 5.2 测试驱动

关键行为都通过 Luajit 测试固定下来：

- UTF-8 文件不转码。
- GB18030 文件可转 UTF-8 后解析。
- 无 `iconv` 时仍使用内置 GB18030 解码。
- 缓存命中时跳过扫描。
- 手工目录存在时插件让位。
- 候选章节不足 3 条时不接管默认目录。

### 5.3 设备闭环

项目不是只在本地测试完成，而是接入 Kindle 挂载卷做验证：

- 对比本地插件和 Kindle 插件 checksum。
- 读取 Kindle 上真实 TXT 文件测试解析数量。
- 检查 KOReader metadata 中的插件状态。
- 将修复后的 Lua 文件同步到 Kindle 插件目录。

## 7. 项目价值总结

这个项目的核心成果不是单个正则或单个按钮，而是把“中文 TXT 小说阅读体验”从不可控的手工操作，变成了可自动识别、可缓存、可重建、可排查、可在 Kindle 实机运行的插件能力。

AI 在项目中的主要价值体现在：

- 将模糊需求拆解成插件生命周期、解析、映射、缓存、编码和设备验证等模块。
- 持续根据真实设备反馈定位根因，而不是停留在本地可运行。
- 用测试锁定每次修复，减少后续回归。
- 在短周期内完成从项目创建、功能迭代、实机排障到文档交付的闭环。
